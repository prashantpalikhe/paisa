### WebAuthn Concepts

WebAuthn (Web Authentication) is a W3C standard for passwordless authentication using public-key cryptography.

**Two ceremonies:**

1. **Registration** (adding a passkey to an existing account):
   - Server generates a challenge + config (`generateRegistrationOptions`)
   - Browser prompts biometric via `navigator.credentials.create()`
   - Browser creates a keypair, signs the challenge, sends response back
   - Server verifies signature, stores the public key in DB

2. **Authentication** (logging in with a passkey):
   - Server generates a challenge (`generateAuthenticationOptions`)
   - Browser prompts biometric via `navigator.credentials.get()`
   - Browser signs the challenge with the stored private key
   - Server verifies signature against the stored public key
   - No email/password needed - the authenticator holds the credential

**Key terms:**
- **RP (Relying Party)**: Our app, identified by `rpId` (domain) and `rpName`
- **Credential ID**: Unique identifier for a passkey, stored as base64url
- **Challenge**: Random nonce, single-use, 60-second TTL
- **Counter**: Incremented on each use (clone detection)
- **Discoverable credential**: Passkey that lives on the device without needing to identify the user first

## Architecture Decisions

### 1. Challenge Store: In-Memory Map (not Redis)

**Decision:** Store WebAuthn challenges in an in-memory `Map<string, StoredChallenge>`.

**Why:** Redis is planned for Phase 8. Challenges are short-lived (60s), single-use, and self-cleaning. Acceptable tradeoff for single-server deployments.

**Safeguards:**
- 60-second TTL per challenge
- Periodic cleanup every 60 seconds via `setInterval` (cleaned up in `onModuleDestroy`)
- `MAX_CHALLENGES = 10,000` cap with FIFO eviction of 100 oldest when full
- Single-use: deleted immediately after consumption

**Limitation:** Won't work across multiple server instances. Same limitation as email verification tokens.

### 2. Opaque Session ID for Authentication Challenges

**Decision:** Key authentication challenges by a random hex `sessionId`, not the challenge value itself.

**Why:** The challenge value is sent to the browser. Using it as a lookup key means the browser controls which challenge gets consumed. An opaque session ID returned alongside the options prevents the browser from targeting arbitrary challenges.

**Flow:** `generateAuthenticationOptions()` returns `{ ...options, sessionId }`. The frontend sends `{ response, sessionId }` to `login/verify`.

### 3. hasPasskey NOT Queried on Every Request

**Decision:** `JwtStrategy.validate()` hardcodes `hasPasskey: false`. Real value is only populated by `GET /auth/me` and login/register responses.

**Why:** The original implementation injected `PasskeyService` into `JwtStrategy` and ran `hasPasskey(userId)` on every authenticated request - an extra DB query on every single API call. This is a performance disaster at scale.

**Tradeoff:** `@CurrentUser().hasPasskey` is always `false` in arbitrary endpoints. Only `GET /auth/me` returns the real value. The frontend calls `GET /auth/me` on load and after passkey changes.

### 4. Feature Flag Guard at Controller Level

**Decision:** `@UseGuards(PasskeyGuard)` applied to the entire `PasskeyController` class, not individual routes.

**Why:** Same pattern as `GoogleOAuthGuard`. All 7 passkey endpoints should be invisible when the feature is disabled. Controller-level guard means new endpoints are automatically gated.

### 5. Shared Auth Helpers

**Decision:** Extract `setRefreshCookie()`, `toAuthUser()`, `parseExpiryToMs()` into `auth.helpers.ts`.

**Why:** Both `AuthController` and `PasskeyController` needed identical cookie-setting and user-mapping logic. The original code duplicated ~60 lines of private methods.

### 6. Set-Password vs Change-Password

**Decision:** Separate `POST /auth/set-password` endpoint (no current password required) from existing `POST /auth/change-password` (requires current password).

**Why:** OAuth-only users have `passwordHash: null`. They can't provide a "current password" because they don't have one. Security is maintained because they're already authenticated via JWT (which required OAuth login).

**Guard:** `ConflictException` (409) if user already has a password. Prevents abuse.

## Challenges Encountered

### 1. PrismaService vs DatabaseService

**Problem:** Initially imported `PrismaService` in `PasskeyService`, which doesn't exist.

**Fix:** The codebase uses `DatabaseService` (which extends `PrismaClient`). Had to search for the actual class name and correct the import.

### 2. credentialId Must Be Valid base64url

**Problem:** E2e test helper generated credentialId as `cred-${Date.now()}-${Math.random()}`. `@simplewebauthn/server` v13 strictly validates credential format and threw 500 errors.

**Fix:** Generate proper random bytes and encode as base64url:
```typescript
const randomBytes = Buffer.from(
  Array.from({ length: 32 }, () => Math.floor(Math.random() * 256)),
);
const credentialId = randomBytes
  .toString('base64')
  .replace(/\+/g, '-')
  .replace(/\//g, '_')
  .replace(/=/g, '');
```

### 3. HTTP Status Code 201 vs 200

**Problem:** NestJS POST endpoints default to 201 Created. Tests expected 200.

**Fix:** Added `@HttpCode(HttpStatus.OK)` to registration/authentication endpoints. These aren't "creating a resource" in the REST sense - they're performing a ceremony.

### 4. Unicode Characters in Edit Operations

**Problem:** The `━` box-drawing characters used in section headers got corrupted during Edit tool string matching.

**Fix:** Used different anchor strings that didn't include Unicode characters, or read the exact lines first.

### 5. WebAuthn Ceremonies Can't Be E2E Tested

**Problem:** Full registration/authentication flows require a real browser authenticator (Touch ID, etc.). Supertest can't simulate this.

**Fix:** Test everything around the ceremonies:
- Options generation (challenge, config, excludeCredentials)
- CRUD operations (list, rename, delete with DB-seeded passkeys)
- Auth requirements (401 without JWT)
- Cross-user isolation (404 for other user's passkeys)
- Feature flag gating
- `hasPasskey` flag enrichment on `/auth/me`

The actual crypto verification is tested internally by `@simplewebauthn/server`.

### 6. Pre-existing Google OAuth Test Failures

**Problem:** 2 tests expect 404 when Google OAuth disabled, but `.env` has `FEATURE_AUTH_GOOGLE_ENABLED=true`.

**Status:** Deferred. These tests need env var mocking or a test-specific `.env`.

| Severity | Issue | Details |
|----------|-------|---------|
| **HIGH** | Registration challenge keyed by userId | If same user opens two tabs, second call overwrites first challenge. Acceptable for now but could add random suffix. |

| **MEDIUM** | `PasskeyInfo.deviceName` frontend type says `string | null` | Verify Prisma schema matches. Backend may always provide a default via `inferDeviceName()`. |
